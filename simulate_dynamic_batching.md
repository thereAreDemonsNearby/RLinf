# Background
1. 场景描述：有一堆GPU卡（如20张卡），还有一堆机器人（如20个机器人）。我们想要用这些卡来驱动这些机器人。一张卡进行一次推理，能产生一个动作块，一个动作块包含30个动作，然后机器人从GPU接收到这个动作块之后，需要依次执行这30个动作。机器人执行完30个动作之后，需要将它的最新观察（基本上是几张图片）发回给某一张GPU，然后这个GPU进行下一轮推理。
2. 推理的模型：一个模型由vision_encoder、language_model、denoise 三个模块组成，这三个模块依次执行。
2. 耗时数据：机器人执行一个动作的时间在35ms到45ms之间浮动，并且连续执行完30个动作之后才会停下来。至于一张GPU上的推理时间，有一些实测的数据，见 speed_pi05.csv，这个表格展示了不同的batch_size下，整个网络的执行时间（total_time）、vision_encoder的执行时间（vision_time）、language_model的执行时间（lm_time）还有denoise的执行时间（denoise_time）。一共测了batch_size为1、2、4、8、16、32这5中情况。如果需要其他batch size下的耗时情况，可以根据已有的数据进行差值。

# Task
现在需要开发一个模拟器，用于估计GPU的负载情况和机器人的忙碌和等待情况。GPU的卡数和机器人的数量都需要是可以调的。模拟的细节如下：
1. 初始状态下，所有机器人都是静止的，所有GPU都是空闲的。
2. 首先机器人们将他们的第一个观察（图片）发送给相对比较空闲的一张GPU卡。发送观察图片的网络延迟需要是可配置的。
3. 一张GPU卡拿到机器人的观察之后，有权在一定的时间阈值（可配置）之内，将多个观察拼到一个batch里面，然后用这个batch进行推理，拼到的观察的数量即为batch_size（我希望这是个偶数）。batch_size需要有一个上限（可配置），因为batch_size过大会导致推理延迟很高。不同batch_size下的推理延迟见 speed_pi05.csv。
4. 一张GPU对一个batch完成推理之后，将推理结果发给机器人，即每个机器人收到一个30个动作的动作块。发送动作块的延迟需要是可配置的。
5. 然后机器人将收到的这30个动作依次执行，执行完成后就将最新的观察发回给相对比较空闲的一张GPU卡，然后GPU进行下一轮推理。
6. 一次推理+一次机器人执行称为一个round。周而复始。模拟的round数需要是可配置的。

你不但需要模拟这个过程，还需要统计模拟过程中的GPU利用率、机器人执行两个动作块之间等待GPU推理的延迟（用P50、P90这样的分位数来表示）。最好还需要将模拟出来的过程用html画出来，让我能一目了然。
